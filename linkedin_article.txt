I worked on IoT hardware for three years. Flow meters, level sensors, temperature probes, RS485 Modbus sensors wired into field devices across East Africa. When something needed debugging at the hardware level, the options were always the same: QModMaster on a Windows laptop, or a throwaway Python script with pymodbus boilerplate.

Both are painful when you just want to check a register value quickly.

I kept thinking: why isn't there a curl for Modbus? One command, one line, done. So after I left that job, I built it.

What modbus-cli does

pip install modbus-curl

Then:

modbus read 192.168.1.10 40001 --count 10
modbus write 192.168.1.10 40001 1234
modbus scan 192.168.1.10 --range 1-10
modbus watch 192.168.1.10 40001 --count 8
modbus dump 192.168.1.10 40001 40200 --csv registers.csv

Five commands. That's the entire tool. Read registers, write values, scan for devices on a bus, watch values live, dump a range to CSV or JSON.

For those outside industrial automation: Modbus is a 45-year-old communication protocol that refuses to die. It's how most industrial sensors and PLCs talk to each other. The protocol is simple. The tooling around it is not.

The tool auto-detects register types from standard Modbus addressing. Type 40001 and it knows you want a holding register. Type 30001 and it reads input registers. Anyone who's dealt with the 0-based vs 1-based addressing confusion knows why this matters. modbus-cli handles both.

The watch dashboard

The read/write/scan commands are useful. The watch mode is where it gets interesting.

I built a full-screen terminal dashboard using Textual, the Python TUI framework from the team behind Rich. It gives you a live data table that updates every poll cycle, sparkline history per register showing the last 60 samples, change detection with deltas between polls, and a stats bar tracking poll count and timing.

Press f to cycle through decimal, hex, binary, and signed formats. Press p to pause. Press r to reset.

When a sensor starts drifting, you need to watch the raw register values over time and spot the pattern. Staring at a Python while-loop printing numbers is not the way to do that. The sparklines give you the trend instantly.

Design decisions

TCP and serial RTU in the same tool. I've seen setups where some sites use TCP gateways and others use direct RS485. One tool should handle both. Just add --serial /dev/ttyUSB0 and it switches to RTU mode. Same commands, same output.

Styled terminal output with progress bars. When you're scanning through 247 possible slave IDs on a bus, you want to see devices discovered as they appear, not wait for a wall of text at the end.

CSV and JSON export. modbus dump reads registers in chunks of 125 (the Modbus protocol maximum per request) with a progress bar, and writes everything to a file. Add --json to any command and pipe it into jq or feed it into whatever automation you have.

What happened when I open-sourced it

I pushed the repo on a Saturday. By Monday I had pull requests from two different contributors. One added Docker support. Another added JSON output to every command. A third opened a PR for 32-bit float decoding across register pairs.

I hadn't promoted it anywhere at that point. People found it through GitHub search. That told me there's a gap in the tooling that other people feel too.

  - JSON output (merged): https://github.com/19bk/modbus-cli/pull/8

Testing without hardware

The repo includes a simulator that starts a Modbus TCP server on port 5020 with three slave devices and 100 registers. The values drift every 500ms to simulate real sensor behavior: temperature wanders between 20 and 28 degrees, pressure fluctuates around 1000 mbar, battery voltage slowly drops.

python simulator.py

Then in another terminal:

modbus read localhost 40001 -c 10 -p 5020
modbus watch localhost 40001 -c 8 -p 5020

The drifting values make the watch dashboard sparklines actually move. Worth trying even if you've never touched Modbus before.

What I'm building next

Register map files, so you can see "temperature" instead of "40001". 32-bit float decoding across register pairs, with configurable byte and word order. Modbus ASCII protocol support for older equipment.

If you work with Modbus devices, PLCs, or SCADA systems, give it a try. Feature requests and PRs are welcome.
